API keys let you authenticate requests to Atmet's API from outside the web app — whether that's a CI pipeline, an internal dashboard, a custom script, or a third-party integration. Each key is tied to a specific workspace, so your code can read and trigger automations, retrieve run history, and interact with other Atmet resources on behalf of your team. This page covers how to create, use, and revoke API keys safely.

## Creating an API Key

<Warning>
  The full secret value of an API key is shown **only once**, immediately after creation. Atmet does not store a retrievable copy. If you lose the key before saving it, you'll need to delete it and create a new one.
</Warning>

<Steps>
  <Step title="Open API Keys">
    Go to **Settings → API Keys** (or select **API Keys** from the workspace section of the sidebar).
  </Step>
  <Step title="Click New API Key">
    Click the **New API Key** button to open the creation dialog.
  </Step>
  <Step title="Name your key">
    Give the key a short, descriptive name that identifies where or how it will be used — for example, `CI automation`, `Internal dashboard`, or `Zapier integration`. This name is for your reference only.
  </Step>
  <Step title="Create the key">
    Click **Create**. Atmet generates the key and displays the secret value.
  </Step>
  <Step title="Copy and store the secret">
    Copy the secret immediately and store it in a secure location — your environment variables, a secrets manager, or a password vault. Once you close the dialog, the value cannot be retrieved again.
  </Step>
</Steps>

## Using an API Key

Include your API key as a Bearer token in the `Authorization` header of every request.

```http
Authorization: Bearer YOUR_API_KEY
```

<CodeGroup>

```bash curl
curl https://app.atmetai.com/api/automations \
  -H "Authorization: Bearer YOUR_API_KEY"
```

```typescript TypeScript
const response = await fetch('https://app.atmetai.com/api/automations', {
  headers: {
    'Authorization': `Bearer ${process.env.ATMET_API_KEY}`
  }
});

const data = await response.json();
```

</CodeGroup>

Store the key in an environment variable (such as `ATMET_API_KEY`) rather than hardcoding it in your source files. This keeps the secret out of version control and makes it easy to rotate without changing your code.

## Revoking a Key

If a key is no longer needed or you suspect it has been compromised, revoke it immediately.

1. Go to **Settings → API Keys**.
2. Find the key you want to revoke in the list.
3. Click the delete (🗑) icon next to the key's name.
4. Confirm the deletion.

The key is invalidated instantly. Any requests using that key will begin receiving authentication errors immediately after revocation.

## Listing Your Keys

The API Keys page shows all keys currently active in your workspace, including each key's name and the date it was created. The secret value is never displayed again after the initial creation step — only the key name and metadata are visible.

Use descriptive names when you create keys so you can identify them at a glance without needing to track down which application uses which key.

<Note>
  API keys are scoped to the workspace they were created in. A key generated in workspace A cannot access the resources of workspace B. If you need programmatic access to multiple workspaces, create a separate key in each one.
</Note>

## Key Security Best Practices

<CardGroup cols={2}>
  <Card title="Never commit keys to source control" icon="code-branch">
    Keep API keys out of your codebase. Use environment variables or a secrets manager so keys never appear in Git history.
  </Card>
  <Card title="One key per application" icon="key">
    Create a dedicated key for each application or integration. This limits the blast radius if one key is compromised — you can revoke it without disrupting other services.
  </Card>
  <Card title="Rotate keys regularly" icon="arrows-rotate">
    Periodically replace old keys with new ones, especially for long-running integrations. Revoke the old key only after the new one is in place.
  </Card>
  <Card title="Revoke immediately if compromised" icon="shield-exclamation">
    If you suspect a key has been exposed, revoke it right away from **Settings → API Keys**. Create a new key and update your integration to restore access.
  </Card>
</CardGroup>
